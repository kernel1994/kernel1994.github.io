概述

Spring 是一个开源框架，为了解决企业级应用开发的复杂性而创建。它可以让简单的 JavaBean 实现之前只有 EJB (Enterprise JavaBean) 才能完成的事情。Spring 使用 bean 或者 JavaBean 来表示应用组件，但并不意味着 Spring 组件必须要遵循 JavaBean 规范。一个 Spring 组件可以是任何形式的 POJO (Plain Old Java Object)。
为了降低 Java 开发的复杂性，Spring 采取以下 4 种关键策略：

  • 基于 POJO 的轻量级和最小入侵性编程
  • 通过依赖注入 (Dependency Injection, DI) 和面向接口实现松耦合
  • 基于切面和惯例进行声明式编程
  • 通过切面和模板减少样式代码

装配 Bean

创建应用对象之间写作关系的行为称为装配 (wiring)，这也是依赖注入的本质。
SPring 提供了 3 种装配机制：

  • 显式的 XML 配置
  • 显式的 Java 配置
  • 隐式的 bean 发现机制和自动装配

三种装配方式和以相互搭配使用，但是我推荐尽可能使用自动装配，但是如果有些时候，比如源代码不是由你来维护，这时候就必须使用显式配置，此时我推荐使用JavaConfig，最后如果无法使用 JavaConfig 时才使用 XML 配置方式。

自动装配

Spring 从两个角度实现自动化装配：

  • 组件扫描 (component scanning)：Spring 会自动发现应用上下文中所创建的 bean
  • 自动装配 (autowiring)：Spring 自动满足 bean 之间的依赖

bean 作用域

Spring 应用上下文中所有 bean 都是单例的。单例并不适用于所有场景，于是有其他可选：

  • 单例 (Singleton) ：在整个应用中，只创建 bean 的一个实例
  • 原型 (Prototype) ：每次注入或者通过 Spring 应用上下文获取的时候，都会创建一个新的 bean 实例
  • 会话 (Session) ：在 Web 应用中，为每个会话创建一个 bean 实例
  • 请求 (Request) ：在 Web 应用中，为每个请求创建一个 bean 实例

设想购物车的场景

注意这里还有一个 proxyMode 属性，它解决的是会话或者请求作用域 bean 注入到单例 bean 的问题。上面例子中，ShoppingCart 是会话作用域，而 StoreService 是单例的，会在 Spring 应用上下文加载的时候被创建，但是此时并没有 ShoppingCart。而且系统有多个 ShoppingCart 实例，每个用户一个，我们并不想注入固定的，而是当 StoreService 处理购物车时所使用的 ShoppingCart 正好是当前会话那个。

使用 proxyMode ，Spring 并不会实际的 ShoppingCart bean 注入到 StoreService 中，Spring 会注入一个到 ShoppingCart bean 的代理，这个代理会暴露与 ShoppingCart 相同的方法，所以 StoreService 就会认为它是一个购物车。但是当 StoreService 调用 ShoppingCart 的方法时，代理会对其进行懒解析并将调用委托给会话作用域内真正的 ShoppingCart bean。

如果 ShoppingCart 是一个具体的类的话，Spring 就无法创建基于接口的代理了。此时它必须使用 CGLib 来生成基于类的代理。所以，如果 bean 具体类型是具体类的话，我们必须将 ProxyMode 属性设置为 ScopedProxyMode.TARGET_CLASS。
